是的，ccloud_special_l4_deactivate_snat 必须结合 iptables 规则才能让 dig 正常工作

单独配置这个 tag 只完成了 DSR 的"上半段"（LB 不再做 SNAT），但"下半段"（响应包正确返回）完全依赖 Worker 节点上的 iptables + 路由规则。 没有这些规则，DNS 响应包会丢失或被客户端拒绝——这正是你 dig 超时的原因。

为什么 dig 需要 iptables 配合

正常模式下（有 SNAT），LB 全程代理流量，后端不需要任何特殊配置。一旦关闭 SNAT（启用 DSR），响应路径完全改变，Worker 节点必须自行解决三个问题：

问题没有 iptables 会怎样对应 iptables 规则
conntrack 干扰内核自动 reverse-NAT 破坏你的自定义 SNATraw/PREROUTING → NOTRACK
响应包路由走默认路由直接发给客户端，绕过 F5，客户端收到源 IP 不是 LB VIP 的响应并丢弃mangle/PREROUTING → MARK 0x1 + 路由表 100
源 IP:Port 匹配F5 无法关联响应到其会话表nat/POSTROUTING → SNAT 到 NODE_IP:NODEPORT

完整的 DSR 数据包流（来自 entrypoint.sh）

entrypoint.sh ↗ 中 S4 DNS View 自动配置（L229-402）创建了三条 iptables 规则和一条路由：

规则 1：禁用连接跟踪（L339-350）

Bash
iptables -t raw -A PREROUTING -p udp -d 100.64.0.0/12 --dport 53 \
    -j NOTRACK -m comment --comment "S4_DNSVIEW_CC"

作用：对发往 Pod 网段（100.64.0.0/12 ）53 端口的 UDP 包跳过 conntrack。如果不跳过，内核会自动 reverse-NAT 响应包，覆盖掉你后面配置的自定义 SNAT。

规则 2：标记响应包（L352-364）

Bash
iptables -t mangle -A PREROUTING -p udp -s 100.64.0.0/12 --sport 53 \
    ! -d 100.64.0.0/12 \
    -j MARK --set-mark 1 -m comment --comment "S4_DNSVIEW_CC"

作用：对从 Pod 发出的 DNS 响应（源端口 53，目标不是 Pod 网段）打上 fwmark 0x1 标记，使其走路由表 100 而非默认路由。

路由表 100：强制响应包经过 F5（L289-311）

Bash
ip rule add fwmark 0x1 lookup 100
ip route replace default via <LB_VIP> dev <interface> table 100

作用：被标记的响应包走路由表 100，默认网关指向 LB VIP。这确保响应必须经过 F5，由 F5 将源 IP 改回 LB VIP 后再发给客户端。

规则 3：SNAT 源地址（L366-378）

Bash
iptables -t nat -A POSTROUTING -p udp -s 100.64.0.0/12 --sport 53 \
    ! -d 100.64.0.0/12 \
    -j SNAT --to-source <NODE_IP>:<NODEPORT> -m comment --comment "S4_DNSVIEW_CC"

作用：将响应包的源从 POD_IP:53 改为 NODE_IP:NODEPORT ，让 F5 能匹配到其会话表（F5 记录的是 NODE_IP:NODEPORT ）。

完整数据包流

plaintext
DNS 查询入站：
Client ──[dst=LB_VIP:53]──> F5(DSR) ──[dst=NODE:NODEPORT, src保留CLIENT_IP]──> Worker
 
Worker 节点处理：
  raw/PREROUTING  → NOTRACK (跳过conntrack)
  kube-proxy DNAT → NodePort 转发到 Pod
  Pod 处理 DNS 查询 → 生成响应

DNS 响应出站：
  mangle/PREROUTING → MARK 0x1 (标记响应包)
  路由决策          → fwmark 0x1 → 查路由表 100 → 网关=LB_VIP
  nat/POSTROUTING   → SNAT src 改为 NODE_IP:NODEPORT
  Worker ──[src=NODE:NODEPORT, dst=CLIENT_IP]──> F5 ──[src改为LB_VIP:53]──> Client

如果缺少这些规则中的任何一条，响应包要么走错路径，要么源 IP 不匹配，要么被 conntrack 干扰——结果都是 dig 超时。

你需要做什么

部署 k8s-cust-node ↗ Helm Chart 作为 DaemonSet，并在 ConfigMap 中正确配置参数：

  • S4_DNSVIEW_SCI_AUTO_CONFIG=true （自动模式，适用于 DNS 场景）
  • 或者在 iptables-cm 和 routetab-cm 中手动定义上述三条规则和路由

配置完成后 initContainer 会在每个 Worker 节点上自动应用这些规则，之后 dig 就能正常工作。
