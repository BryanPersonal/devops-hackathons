Yes, ccloud_special_l4_deactivate_snat Requires iptables Rules for dig to Work

Configuring the tag alone only completes the "first half" of DSR (LB stops performing SNAT). The "second half" (ensuring response packets return correctly to the client) depends entirely on iptables + routing rules on the Worker nodes. Without these rules, DNS responses are either lost or rejected by the client -- which is exactly why your dig times out.

Why dig Needs iptables

In normal mode (with SNAT), the LB proxies traffic end-to-end, and the backend needs no special configuration. Once SNAT is disabled (DSR enabled), the response path changes completely. The Worker node must solve three problems on its own:

The Three iptables Rules (from entrypoint.sh)

The S4 DNS View auto-configuration in entrypoint.sh ↗ (L229-402) creates three iptables rules and one route:

Rule 1: Disable Connection Tracking (L339-350)

Bash
iptables -t raw -A PREROUTING -p udp -d 100.64.0.0/12 --dport 53 \
    -j NOTRACK -m comment --comment "S4_DNSVIEW_CC"

Purpose: Skip conntrack for UDP packets destined to the Pod subnet (100.64.0.0/12 ) on port 53. Without this, the kernel automatically reverse-NATs response packets, overriding your custom SNAT rule below.

Rule 2: Mark Response Packets (L352-364)

Bash
iptables -t mangle -A PREROUTING -p udp -s 100.64.0.0/12 --sport 53 \
    ! -d 100.64.0.0/12 \
    -j MARK --set-mark 1 -m comment --comment "S4_DNSVIEW_CC"

Purpose: Tag DNS responses coming from Pods (source port 53, destination outside Pod subnet) with fwmark 0x1 , forcing them into routing table 100 instead of the default route.

Routing Table 100: Force Responses Through F5 (L289-311)

Bash
ip rule add fwmark 0x1 lookup 100
ip route replace default via <LB_VIP> dev <interface> table 100

Purpose: Marked response packets use routing table 100, whose default gateway points to the LB VIP. This ensures responses must pass through F5, which rewrites the source IP back to the LB VIP before delivering to the client.

Rule 3: SNAT Source Address (L366-378)

Bash
iptables -t nat -A POSTROUTING -p udp -s 100.64.0.0/12 --sport 53 \
    ! -d 100.64.0.0/12 \
    -j SNAT --to-source <NODE_IP>:<NODEPORT> -m comment --comment "S4_DNSVIEW_CC"

Purpose: Rewrite the response source from POD_IP:53 to NODE_IP:NODEPORT , so F5 can match it against its session table (F5 records the backend as NODE_IP:NODEPORT , not the Pod IP).

Complete Packet Flow

plaintext
DNS Query (Inbound):
Client ──[dst=LB_VIP:53]──> F5(DSR) ──[dst=NODE:NODEPORT, src=CLIENT_IP]──> Worker
 
Worker Node Processing:
  raw/PREROUTING  → NOTRACK (skip conntrack)
  kube-proxy DNAT → Forward NodePort to Pod
  Pod processes DNS query → Generates response

DNS Response (Outbound):
  mangle/PREROUTING → MARK 0x1 (tag response packet)
  Routing decision  → fwmark 0x1 → lookup table 100 → gateway = LB_VIP
  nat/POSTROUTING   → SNAT src to NODE_IP:NODEPORT
  Worker ──[src=NODE:NODEPORT, dst=CLIENT_IP]──> F5 ──[src rewritten to LB_VIP:53]──> Client

If any one of these rules is missing, response packets either take the wrong path, have a mismatched source IP, or get corrupted by conntrack -- all resulting in dig timeout.

What You Need to Do

Deploy the k8s-cust-node ↗ Helm Chart as a DaemonSet, with correct parameters in the ConfigMap:

  • Automatic mode (for DNS scenarios): Set S4_DNSVIEW_SCI_AUTO_CONFIG=true -- the script auto-discovers the LoadBalancer IP, NodePort, and node IP, then creates all rules automatically
  • Manual mode: Define the three rules and route explicitly in the iptables-cm and routetab-cm ConfigMaps

Once deployed, the initContainer applies these rules on every Worker node. After that, dig will work correctly.
