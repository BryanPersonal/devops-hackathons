## 一个关键分界线（非常重要）

> **AI 擅长“局部正确”，不擅长“系统正确”。**

你可以放心让 AI 做的事：

* 补全代码
* 改语法错误
* 写 CRUD
* 改 panic / nil pointer
* 写一段 handler / middleware
* 改测试、补 mock
* 翻 API 用法

你 **不能完全外包** 给 AI 的事：

* ❌ 并发模型是否安全（尤其是 Go）
* ❌ 性能瓶颈在哪（alloc / GC / lock）
* ❌ 这个 abstraction 会不会 6 个月后拖垮系统
* ❌ 这个 retry / timeout 会不会在生产雪崩
* ❌ Kubernetes / 云原生场景下的真实行为

👉 **这些问题，AI 会“看起来很自信地胡说八道”。**

## ✅ 现在真正“值钱的深入”

### **1️⃣ Go 的并发与内存模型（这是护城河）**

AI **非常容易写出：**

* 看似没问题
* 实际有 data race / goroutine leak
* 在高并发下慢慢把系统拖死的代码

你需要理解的是：

* `goroutine` 的真实成本
* `channel` 什么时候该用，什么时候是反模式
* `context` 取消是**逻辑语义**不是装饰品
* `sync.Once / Mutex / RWMutex` 的真实代价
* GC 对延迟的影响（尤其在云原生）

👉 **这是 Cursor 修不了的 bug 类型。**

---

### **2️⃣ 标准库“设计哲学”，而不是用法**

不是记 API，而是理解：

* 为什么 Go 的 error 是这样设计的
* 为什么没有 exceptions
* 为什么 interface 是隐式实现
* 为什么 `http.Handler` 这种接口能撑 10 年

你一旦理解这些：

> AI 给你的代码，你一眼就能判断：
> **“这段 Go 味儿不对”**

---

### **3️⃣ 架构判断力（AI 最弱的一环）**

AI 能生成：

* 一个能跑的 service

但你要判断：

* 这个 service **是否该拆**
* 这个抽象 **是不是提前设计**
* 这个 retry / backoff **会不会压垮下游**
* 这个 client **是否该共享**
* 这个 cache **是不是假优化**

👉 这不是语言问题，是**工程判断力**
但它**必须建立在你真的懂这门语言如何在生产环境中行为**。

## 一个非常实用的学习策略（Vibe Coding 时代版）

### **“反向学习法”**（我强烈推荐你用）

1. 用 Cursor / AI 写代码（完全 OK）
2. **当它给你一段代码时，问自己 3 个问题：**

   * 这段代码在并发下安全吗？
   * 这段代码在失败时行为可控吗？
   * 这段代码 6 个月后还敢改吗？
3. **只对答不上来的地方深入**

   * 查 runtime / net/http / sync
   * 用 `pprof` / `race` 去验证
4. 不追求“全懂”，追求**“关键路径全懂”**

> 你不是在学 Go
> 你是在**训练自己的工程直觉**
