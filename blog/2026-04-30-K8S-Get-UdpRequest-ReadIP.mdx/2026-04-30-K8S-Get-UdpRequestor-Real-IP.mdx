Preserving the client source IP in kubernetes is critical for security, logging, auditing/compliance and application logics. 
By default, many kubernetes networking configurations (like `service` type `NodePort` or `LoadBalancer`) perform Source NAT (SNAT), which replaces the original client IP with the IP of worker node.

In a typical Kubernetes cluster using Calico CNI and kube-proxy (iptables mode), an UDP request from a LoadBalancer to a backend Pod undergoes one DNAT and one SNAT by default, resulting in two NAT operations. If externalTrafficPolicy is set to Local, SNAT is avoided, reducing it to a single DNAT.

{/* truncate */}

> **DNAT is applied in the PREROUTING chain before routing decisions are made, typically affecting incoming traffic. SNAT is applied in the POSTROUTING chain after routing, modifying packages as they leave the nodes. In Kubernetes, a single incoming request can experience both DNAT (on ingress) and SNAT(on egress) within the same node.


| NAT type    | Where it happens (iptables) | From a “traffic flow” perspective |
| ----------- | ----------------------------- | --------------------------------- |
| **DNAT**    | PREROUTING / OUTPUT           | 👉 Ingress (as traffic enters the node) |
| **SNAT**    | POSTROUTING                   | 👉 Egress (as traffic leaves the node)  |

> Now it's time to figure out PREROUTING and POSTROUTING in iptables. 😊

An flow example:
> UDP Traffic: CLinet -> LB - > Node(IP:NodePort) -> kube-proxy -> Pod (Calico CNI)

In this setup, all incoming packets traverse **cali-PREROUTING** first for Calico's pre-processing, then continue to **KUBE-SERVICES** where kube-proxy performs service matching and DNAT. Equal counters indicate no early packet drops or bypass before kube-proxy.

:::info
The rules implement a two-stage NAT pipeline: first, kube-proxy matches the ClusterIP and selects a backend Pod using probabilistic load balancing via KUBE-SVC and KUBE-SEP chains. Then, DNAT rewrites the destination IP to the selected Pod. In parallel, KUBE-MARK-MASQ marks traffic originating outside the cluster CIDR for SNAT, ensuring symmetric return paths via MASQUERADE in POSTROUTING.
:::

#### How Client Source IP plays a role while decommissioning a Core Service running in enterprise infrastructure ?
Preserving the client source IP a critical aspect when decommissioning a service running on a traditional VM. The decommissioning process is typically carried out in several phases:
1. Notify stakeholders in advance and request them to stop using the service
2. Monitor incoming traffic to the service
3. Identify any remaining client source IPs still accessing the service and proactively reach out to those users.
4. Shut down the service for a grace period (e.g., two weeks) to ensure no dependencies remain.
5. Fully decommission the service

Typically, software provides its own mechanisms to capture and log the client's source IP address - for example, BIND and 389 Directory Server both include native capabilities to record client connection details. From a technical pespectives, this works across several layers:
1. Transport / socket layer (L3/L4)
At the lowest lower, the application obtains the client IP from TCP/UDP socket. When a client connects, the kernel exposes the peer address via system calls like `accept()` (TCP) or `recvfrom()`(UDP). 
- In DNS (BIND), each query arrives over UDP or TCP, and the service logs the source IP directly from the packet metadata.
- In LDAP (389ds), the server retrieves the client IP from the established TCP session.

:::warning

Those approaches are most reliable, as long as the source IP remains unmodified by intermediate systems. In reality, however, this is rarely possible due to pervasive NAT.
While preserving source IP via `externalTrafficPolicy: Local`, please be aware of [Caveats and Limitations](https://kubernetes.io/docs/tasks/access-application-cluster/create-external-load-balancer/#caveats-and-limitations-when-preserving-source-ips)
:::
